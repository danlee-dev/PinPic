# 관리자 패널 수치 레퍼런스

관리자 패널(관리 탭)에서 보이는 모든 수치가 정확히 무엇을 의미하는지, 어디서 집계되는지, 해석할 때 주의할 점은 무엇인지 정리한 문서.

---

## 탭 구조

```
관리 탭
├── 대기 (승인 대기 사진 개수)
├── 사진 (전체 사진 관리)
├── 관심도
│   ├── 투표 (투표 단계 조회/클릭)
│   ├── 결과 (결과 발표 단계 페이크 도어)
│   └── 분석 (투표+결과 종합 분석)
├── 원본 순위 (조작 전 진짜 votes 기준)
└── 설정 (투표 기간 / 결과 발표 시각 / 미리보기 / 관리자 계정)
```

---

## 1. 관심도 > 투표

투표 진행 중에 사용자들이 사진을 얼마나 보고, 촬영 스팟 정보에 얼마나 관심을 보였는지 측정한다. 모든 수치에서 **관리자의 활동은 포함되지 않는다** (어드민 필터링 적용).

### 1.1 전체 통계

| 수치 | 의미 | DB 기준 |
|---|---|---|
| **사진 조회** | photo modal과 무관하게 사진 카드가 사용자의 화면에 **노출된** 총 횟수 (세션 내 dedup). 같은 사진을 두 번 보면 1회로 카운트. | `photo_views` 테이블 row 수 |
| **스팟 클릭** | photo modal 안에서 "촬영 스팟 및 설정값 보기" 티저 버튼을 누른 횟수. 투표 단계에서만 발생 (결과 발표 후에는 이 버튼이 사라짐). | `photo_clicks` 테이블 row 수 |
| **전환율** | `스팟 클릭 / 사진 조회 * 100` (%). 사진을 본 사람 중 촬영 정보까지 궁금해한 비율. |
| **사진 열람 N명** | 한 번이라도 사진 카드가 노출된 unique 사용자 수 (로그인 기준). | `photo_views` distinct `user_id` |
| **스팟 클릭 N명** | 한 번이라도 스팟 버튼을 누른 unique 사용자 수. | `photo_clicks` distinct `user_id` |
| **스팟 미클릭 N명** | `사진 열람 - 스팟 클릭` 유저. 사진은 봤지만 정보까지는 안 궁금해한 사람. |
| **로그인 (조회/클릭)** | 로그인 상태에서 발생한 조회/클릭 수. | `user_id IS NOT NULL` |
| **비로그인 (조회/클릭)** | 비로그인(익명) 상태에서 발생한 조회/클릭 수. | `user_id IS NULL` |

### 1.2 사진별

각 출품작(사진)마다:

| 컬럼 | 의미 |
|---|---|
| **닉네임** | 출품자 닉네임 |
| **N명/N명** | unique viewer / unique clicker (해당 사진의 고유 열람자 / 고유 클릭자) |
| **N뷰** | 해당 사진의 총 조회 횟수 |
| **N클릭** | 해당 사진의 스팟 버튼 클릭 횟수 |
| **N%** | 전환율 (`클릭/조회*100`) |

**해석**:
- 전환율이 높은 사진 = "어떻게 찍었는지 알고 싶다"는 깊은 호기심 유발
- 조회는 많은데 전환율 0% = 시각적 매력은 있지만 정보 욕구는 안 자극
- 조회는 적은데 전환율 50%+ = 코어 팬이 있는 사진, 정보재 상품 후보

### 1.3 사용자별

각 로그인 사용자마다:

| 컬럼 | 의미 |
|---|---|
| **이메일** | Google OAuth 이메일 (비로그인은 제외됨) |
| **N뷰** | 해당 사용자가 사진을 본 총 횟수 |
| **N클릭** | 스팟 버튼을 누른 횟수 |
| **N%** | 개인별 전환율 |

**해석**:
- 조회가 압도적으로 많은 사용자 = 슈퍼유저. 차후 알파 테스터 후보
- 클릭률이 높은 사용자 = 정보 가치를 높게 평가. 유료 전환 가능성 높은 코호트

## 2. 관심도 > 결과

결과 발표 시각(`reveal_at`) **이후**에 발생한 이벤트만 집계한다. 투표 단계 때 어드민이 테스트한 클릭은 제외되므로 순수한 결과 발표 단계 데이터만 보인다.

### 2.1 결과 발표 단계 한 줄 요약

| 수치 | 의미 |
|---|---|
| **전체 모달 → 사전신청 전환율** | `이메일 제출 완료 / 사진 모달 열기 * 100`. 결과 페이지를 둘러보다가 결국 이메일까지 남긴 사용자의 비율. 가설 검증 핵심 KPI. |
| **사전신청 누적** | waitlist 테이블에 저장된 총 이메일 수 (중복 제외). 절대 WTP 시그널 수. |

### 2.2 단계별 카운터 (6개 카드)

각 카드마다 **총 횟수 / 로그인 수 / 비로그인 수 / 유니크 로그인 수** 4가지를 보여준다.

| 카드 | 의미 | DB 기준 |
|---|---|---|
| **사진 모달 열기** | 사진 카드를 탭해서 photo modal이 열린 횟수. 피드 + 명예의 전당 합계. | `photo_modal_opens` 테이블 전체 |
| **페이크 도어 클릭** | "TOP 10 비밀 전부 열기" 버튼이 클릭된 횟수. photo modal 안, 명예의 전당 카드 안, 상단 CTA bar 모두 포함. | `fake_door_clicks` 중 source에 `__`가 없는 row |
| **상단 CTA 클릭** | 명예의 전당 헤더 위의 큰 입체감 CTA bar를 클릭한 횟수만 따로 분리. | `fake_door_clicks` where `source = 'inline_bar'` |
| **990원 사전신청 클릭** | 페이크 도어 모달이 열리고 "990원에 사전신청 하기" 버튼을 누른 횟수. | `fake_door_clicks` where source에 `__pitch_continue` 포함 |
| **이메일 제출 완료** | 이메일 입력 후 "사전 신청 완료" 버튼을 누른 횟수. | `fake_door_clicks` where source에 `__email_submit` 포함 |
| **waitlist 저장** | 실제로 waitlist DB에 저장된 row 수 (동일 이메일 중복 제외). | `waitlist` 테이블 row 수 |

**로그인 / 비로그인 / 유니크**:
- **로그인 N**: `user_id`가 있는 row 수
- **비로그인 N**: `user_id`가 NULL인 row 수
- **유니크 N**: distinct `user_id` 수 (같은 사람이 여러 번 눌러도 1으로 카운트)

### 2.3 진입 경로별 퍼널 (4종)

source 컬럼의 prefix로 진입 경로를 4가지로 분류하고, 각 경로를 독립된 funnel로 보여준다.

#### 퍼널 A: 피드 → 사진 모달 → 사전신청

```
사진 모달 열기 (피드)     ← photo_modal_opens where source='feed'
  ↓ N%
TOP 10 비밀 클릭          ← fake_door_clicks where source='photo_modal'
  ↓ N%
990원 사전신청 클릭        ← fake_door_clicks where source='photo_modal__pitch_continue'
  ↓ N%
이메일 제출 완료           ← fake_door_clicks where source='photo_modal__email_submit'
```

사용자가 **피드 탭에서** 사진 카드를 탭 → 모달에서 unlock 버튼 클릭 → 모달 진입 → 이메일 제출. 피드에서 시작한 사용자의 행동 흐름.

#### 퍼널 B: 명예의 전당 → 사진 모달 → 사전신청

구조 A와 동일하되, 시작점이 `source='hall_of_fame'`. 결과 발표 컨텍스트에서 시작한 사용자. A와 비교하면 "결과 컨텍스트가 WTP를 자극하는지" 시사.

#### 퍼널 C: 명예의 전당 상단 CTA → 사전신청

```
Hall of Fame 진입 (가정)   ← photo_modal_opens where source='hall_of_fame' (모달 열림을 방문 proxy로 사용)
  ↓ N%
상단 CTA 클릭             ← fake_door_clicks where source='inline_bar'
  ↓ N%
990원 사전신청 클릭        ← fake_door_clicks where source='inline_bar__pitch_continue'
  ↓ N%
이메일 제출 완료           ← fake_door_clicks where source='inline_bar__email_submit'
```

사진을 거치지 않고 **페이지 상단 큰 CTA 버튼**에서 바로 진입한 흐름.

#### 퍼널 D: TOP10 카드 unlock → 사전신청

```
TOP 10 비밀 클릭          ← fake_door_clicks where source starts with 'inline_card_'
  ↓ N%
990원 사전신청 클릭        ← fake_door_clicks where source matches 'inline_card_*__pitch_continue'
  ↓ N%
이메일 제출 완료           ← fake_door_clicks where source matches 'inline_card_*__email_submit'
```

명예의 전당 카드 하단의 인라인 "TOP 10 비밀 전부 열기" 텍스트 클릭에서 시작한 흐름.

#### 퍼널에서 보이는 수치

각 단계 행:
| 요소 | 의미 |
|---|---|
| **숫자 (큰 글씨)** | 해당 단계 총 발생 횟수 |
| **(로 N 비 N)** | 로그인 N건, 비로그인 N건 |
| **N.N%** (색상 코딩) | 직전 단계 대비 전환율. ≥30% 녹색, 15~30% 노랑, <15% 빨강 |
| **파란→빨간 바** | 첫 단계 대비 비율 시각화 |

### 2.4 사진별 (결과 단계)

| 컬럼 | 의미 |
|---|---|
| **학교** | 연세대 / 고려대 |
| **닉네임** | 출품자 |
| **열림** | 해당 사진의 photo modal이 열린 횟수 (`photo_modal_opens`) |
| **클릭** | 해당 사진에서 페이크 도어 버튼이 클릭된 횟수 (`fake_door_clicks` where photo_id 일치, source에 `__` 없음) |
| **제출** | 해당 사진에서 시작하여 이메일 제출까지 완료된 횟수 (`fake_door_clicks` where source ends with `__email_submit`) |

### 2.5 사용자별 (결과 단계, 로그인만)

| 컬럼 | 의미 |
|---|---|
| **이메일** | Google OAuth 이메일 |
| **열림** | 해당 사용자가 photo modal을 연 횟수 |
| **클릭** | 페이크 도어 버튼을 누른 횟수 |
| **제출** | 이메일 제출 완료 횟수 |

### 2.6 사전신청 이메일

waitlist 테이블에 저장된 모든 이메일. 각 row:

| 컬럼 | 의미 |
|---|---|
| **이메일** | 사용자가 직접 입력한 이메일 주소 |
| **소스** | 어떤 진입점에서 사전신청했는지 (`inline_bar` / `photo_modal` / `inline_card_*` 등) |
| **시간** | 제출 시각 (KST 변환 전이라 UTC로 표시될 수 있음) |

### 2.7 최근 페이크 도어 활동

`fake_door_clicks` 테이블의 최근 row를 시간 역순으로 보여주는 raw 로그.

| 컬럼 | 의미 |
|---|---|
| **시간** | 이벤트 발생 시각 |
| **소스** | source 값 그대로 (`inline_bar`, `photo_modal`, `inline_card_3`, `inline_bar__pitch_continue`, `photo_modal__email_submit` 등) |
| **닉네임** | 연결된 사진이 있으면 출품자 닉네임 |
| **로그인/비로그인** | `user_id` 유무 |

**source 값 해석 규칙**:
- `__` 없음 = 첫 클릭 (CTA 버튼 자체)
- `__pitch_continue` = "990원에 사전신청 하기" 버튼
- `__email_submit` = "사전 신청 완료" 버튼
- prefix = 진입 경로 (`inline_bar` / `photo_modal` / `inline_card_1`...`inline_card_10`)

## 3. 관심도 > 분석

투표 단계와 결과 단계를 **연결**해서 보는 종합 분석. 결과 발표 시각 이후 데이터만 집계 (투표 단계 측정과 기간이 겹치지 않음).

### 3.1 가설 검증 한 줄 요약

| 수치 | 의미 |
|---|---|
| **전체 모달 → 사전신청 전환율** | `waitlist 누적 / photo_modal_opens 총 수 * 100`. 사이트에 들어와서 사진을 클릭한 사람 중 최종적으로 이메일까지 남긴 비율. |
| **사전신청 누적** | waitlist 테이블 row 수 (중복 이메일 제외). |

### 3.2 소스별 퍼널 (3종)

결과 탭의 4종 퍼널 중 A(피드), B(명예의전당), C(inline CTA)를 나란히 놓고 비교. 구조는 결과 탭과 동일하되, 3개를 한 화면에서 **비교**하기 위한 용도.

**비교에서 얻는 인사이트**:
- A vs B → 결과 컨텍스트(명예의 전당)가 WTP를 자극하는지 시사
- B vs C → 사진을 보고 들어온 사용자 vs 바로 CTA 누른 사용자 중 누가 진성
- 어느 단계에서 drop-off가 큰지 → 카피/UI 개선 우선순위

### 3.3 투표자 vs 비투표자 (로그인 사용자만)

로그인 사용자를 두 코호트로 나눈다:

| 코호트 | 정의 |
|---|---|
| **투표한 사용자** | `votes` 테이블에 `voter_id`가 해당 유저의 auth uid와 매칭되는 row가 1개 이상 |
| **투표 안 한 사용자** | 위 조건에 해당되지 않지만 `photo_modal_opens` 등에 `user_id`가 존재 |

각 코호트의:
| 수치 | 의미 |
|---|---|
| **N명** | 코호트에 속한 unique 사용자 수 |
| **→ 클릭 N** | 해당 코호트 중 페이크 도어를 1번이라도 클릭한 사용자 수 |
| **N.N%** | 클릭 비율 = 클릭자 / 코호트 인원 |

**주의**: 비로그인 voter_id는 세션 fingerprint라 auth uid와 매핑 불가. 그래서 이 비교는 **로그인 사용자 한정**이다.

**해석**:
- 투표자 클릭률 > 비투표자 → 참여도와 WTP 양의 상관
- 투표자 ≈ 비투표자 → 투표 여부와 무관한 일반적 호기심
- 투표자 < 비투표자 → 비로그인 호기심 그룹이 오히려 결제 의향 → 회원 가입이 결제 마찰일 수 있음

### 3.4 사진별 사전신청 자극도

각 사진에 대해:

| 컬럼 | 의미 |
|---|---|
| **학교** | 연세대 / 고려대 |
| **닉네임** | 출품자 |
| **열림 N** | 해당 사진의 photo modal 열림 횟수 |
| **→ 클릭 N** | 해당 사진에서 페이크 도어 클릭 횟수 |
| **N.N%** | `클릭 / 열림 * 100`. 해당 사진이 얼마나 사전신청 행동을 유발하는지. |

**해석**: 비율이 높은 사진 = 시각적 매력을 넘어 "촬영 노하우가 궁금한" 사진. 작가가 잠재적 콘텐츠 크리에이터 후보.

### 3.5 시간대별 활동 분포

1시간 단위 bucket으로 **KST(한국 시간)** 기준 시간 분포를 보여준다.

각 row:
| 요소 | 의미 |
|---|---|
| **시간** | `YYYY-MM-DD HH:00` (KST) |
| **파란 바** | 해당 시간대의 photo modal 열림 수 (전체 활동 중 상대 비율) |
| **열림 N** | photo modal opens |
| **클릭 N** | 페이크 도어 첫 클릭 (source에 `__` 없는 것) |
| **제출 N** | 이메일 제출 (`__email_submit`) |

**해석**: 결과 발표 직후 피크 → 다음 이벤트 운영 시 이 시간에 집중.

### 3.6 사전신청 이메일

waitlist 테이블에 저장된 모든 이메일을 시간 역순으로. 결과 탭의 것과 동일하지만 분석 화면에서 바로 볼 수 있게.

---

## 4. 원본 순위

### 4.1 원본 인기 순위

`votes` 테이블의 실제 카운트 기준 (vote_overrides 미적용). 일반 사용자에게 보이는 명예의 전당 순위와 **다를 수 있다** — 우리가 표시 목적으로 조작한 offset이 반영되지 않기 때문.

각 사진:
| 컬럼 | 의미 |
|---|---|
| **순위** | 진짜 votes count 기준 내림차순 |
| **학교** | 연세대 / 고려대 |
| **닉네임** | 출품자 |
| **N표** | `votes` 테이블에서 해당 photo_id의 row 수 (진짜 카운트) |
| **+N → N** (노란색) | `vote_overrides` offset이 있는 사진만 표시. `+offset → 표시값`. 어떤 사진이 얼마나 조작됐는지. |

### 4.2 학교 합계 (원본)

연세대 / 고려대 진짜 votes 합계. 표시 순위와 무관한 실 데이터.

## 5. 설정 탭

### 5.1 투표 기간 (KST)

- **시작 / 종료**: datetime-local input. KST 기준으로 표시 + 입력.
- 투표 기간 외에는 사용자의 투표 버튼이 비활성화되고 안내 배지가 뜸.

### 5.2 결과 발표 (KST)

- **발표 시각**: datetime-local input. 이 시각이 지나면 모든 사용자에게 명예의 전당 화면이 자동 활성화.
- 현재 발표 시각이 아래에 표시됨.

### 5.3 화면 미리보기 토글

| 시점 | 토글 라벨 | ON 시 효과 |
|---|---|---|
| 발표 전 | "결과 화면 미리보기" | 어드민에게만 발표 후 화면(Hall of Fame)이 보임 |
| 발표 후 | "투표 중 화면 미리보기" | 어드민에게만 발표 전 화면(투표 중)이 보임 |

localStorage 기반이라 다른 유저에게 영향 없음.

### 5.4 관리자 계정

- 현재 등록된 관리자 이메일 목록
- 이메일로 추가 / 제거
