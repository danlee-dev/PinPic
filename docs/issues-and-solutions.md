# 개발 중 발생한 주요 이슈와 해결 과정

## 1. 페이지네이션 미적용으로 인한 데이터 누락 (반복 발생)

이 문제는 한 번이 아니라 서비스가 성장하면서 **3차례 반복 발생**했다. 매번 다른 곳에서 같은 원인으로 터졌고, 그때마다 대응하면서 점차 체계를 갖춰나갔다.

### 1차: 유저 이메일 조회 누락 (유저 50명 초과 시)

#### 현상

어드민 패널의 관심도 탭에서 사용자 이메일이 일부만 표시되고, 나머지는 `a3f8b2c1...` 같은 UUID 앞 8자리로 표시되었다.

#### 원인

Supabase Auth의 `listUsers()` API가 기본적으로 한 페이지에 50명만 반환한다. 유저가 50명을 넘어가면서 나머지 유저의 이메일을 가져오지 못했다.

```ts
// 수정 전: 첫 50명만 반환
const { data: { users } } = await service.auth.admin.listUsers();
```

#### 해결

page 파라미터를 사용한 페이지네이션을 적용하여 모든 유저를 순회하도록 수정했다.

```ts
// 수정 후: 1000명씩 전체 순회
while (true) {
  const { data: { users } } = await service.auth.admin.listUsers({ page, perPage: 1000 });
  // ... 매핑 처리
  if (users.length < perPage) break;
  page++;
}
```

### 2차: 관심도 통계 쿼리 누락 (사진/유저 증가 시)

#### 현상

어드민 패널에서 사진별/사용자별 관심도 통계가 일부만 표시되고, 실제 데이터와 불일치했다.

#### 원인

관심도 통계를 가져오는 쿼리가 `photo_views`, `photo_clicks` 테이블을 직접 조회하고 있었는데, 이 역시 Supabase 기본 limit 1000행에 걸렸다.

#### 해결

집계 로직 자체를 클라이언트에서 DB로 이전했다. `engagement_totals` 뷰와 `engagement_by_photo` 뷰, `engagement_by_user` RPC 함수를 만들어 서버 사이드에서 집계하도록 변경했다. 이렇게 하면 원본 행 수와 무관하게 집계된 결과만 가져오므로 페이지네이션 문제 자체가 발생하지 않는다.

### 3차: 스프레드시트 집계 누락 (조회수 1000 초과 시)

#### 현상

사이트 어드민 패널에서는 총 조회수가 1300+회로 표시되는데, Google Spreadsheet 집계에서는 1000회로 고정되었다. 스팟 클릭 수는 계속 증가하는데 조회수만 멈춰 있는 상황이었다.

#### 원인

Supabase REST API는 기본 응답 limit이 **1000행**이다. Google Apps Script의 `fetchTable` 함수가 페이지네이션 없이 단일 요청만 보내고 있었기 때문에, `photo_views`가 1000행을 초과하는 시점부터 나머지 데이터가 잘려나갔다.

```js
// 수정 전: 1000행까지만 반환
var url = SUPABASE_URL + "/rest/v1/" + table + "?select=" + select;
```

#### 해결

offset 기반 페이지네이션을 적용하여 1000행씩 반복 조회하고, 마지막 페이지(반환 행 수 < 1000)에서 멈추도록 수정했다.

```js
// 수정 후: 전체 데이터를 1000행씩 순회
while (true) {
  var url = SUPABASE_URL + "/rest/v1/" + table + "?select=" + select
    + "&limit=" + pageSize + "&offset=" + offset;
  // ... fetch & concat
  if (rows.length < pageSize) break;
  offset += pageSize;
}
```

### 잠재적 위험이 남아있는 곳

| 위치 | 현재 상태 | 위험 조건 |
|---|---|---|
| analytics-sync.js `fetchUserEmails` | per_page=1000 고정 | 유저 1000명 초과 시 이메일 누락 |

### 교훈

- Supabase, Firebase 등 대부분의 REST API/Auth API는 기본 응답 제한이 있다 (50행, 1000행 등)
- 개발 초기에 데이터가 적을 때는 문제가 드러나지 않고, 서비스가 성장하면서 갑자기 발생한다
- 같은 원인의 문제가 다른 곳에서 반복적으로 터졌다 -> 한 곳에서 발견하면 전체 코드베이스를 점검해야 한다
- 가능하면 집계를 DB 레벨(뷰, RPC)에서 처리하여 페이지네이션 문제 자체를 회피하는 것이 근본적 해결책이다
- 데이터 정합성 검증 (사이트 vs 외부 집계)을 주기적으로 해야 이런 문제를 조기에 발견할 수 있다

## 2. 이미지 로딩 성능 최적화

### 2-1. 원본 이미지 직접 로딩으로 인한 피드 로딩 지연

#### 현상

초기에는 피드의 모든 사진이 원본 이미지(2~5MB)를 직접 로딩했다. 사진이 40장 이상이면 한 번에 100MB 이상의 트래픽이 발생하여 모바일 환경에서 피드 로딩이 매우 느렸다. 특히 LTE 환경에서는 사진이 하나씩 천천히 뜨면서 사용자 이탈이 발생했다.

#### 해결

썸네일/원본 2단계 로딩 구조를 도입했다.

- 업로드 시점: Google Apps Script에서 원본과 별도로 800px 너비 썸네일(JPEG)을 생성하여 Supabase Storage에 함께 저장
- 피드/목록: `thumb_url`(썸네일)을 사용하여 빠르게 로딩
- 모달(상세 보기): 사진을 눌렀을 때만 `image_url`(원본)을 로딩

```tsx
// 피드 카드: 썸네일 사용
<img src={entry.thumb_url || entry.image_url} />

// 모달: 원본 사용
<img src={entry.image_url} />
```

효과:

- 피드에서 로드하는 이미지 용량이 사진당 2~5MB -> 100~200KB로 감소 (약 95% 절감)
- 모바일 LTE 환경에서도 피드가 빠르게 로딩
- 원본은 상세 보기에서만 로드하므로 고화질 감상에는 영향 없음

### 2-2. 포스터 이미지 미압축 (4.5MB)

#### 현상

피드 상단의 포스터 배너 이미지가 4.5MB PNG 파일이었다. 모든 유저가 피드에 접속할 때마다 이 이미지를 다운로드해야 했고, 특히 첫 방문 시 포스터만으로 수 초의 로딩 시간이 발생했다.

#### 해결

PNG를 WebP로 변환하고 압축하여 4.5MB -> 350KB로 줄였다 (약 92% 절감). `<picture>` 태그로 WebP를 우선 로드하고, 미지원 브라우저에서는 압축된 PNG로 fallback하도록 했다.

```html
<picture>
  <source srcSet="/poster.webp" type="image/webp" />
  <img src="/poster.png" alt="제1회 사진 고연전" />
</picture>
```

### 교훈 (이미지 최적화)

- 모바일 중심 서비스에서 이미지 용량은 곧 사용자 경험이다
- 썸네일/원본 분리는 가장 기본적이면서 효과가 큰 최적화다
- 정적 에셋(포스터, 로고 등)도 형식 변환(WebP)과 압축만으로 큰 개선을 얻을 수 있다
- 처음부터 적용하지 않으면 서비스 운영 중에 성능 이슈로 발견하게 된다

## 3. 피드 정렬 알고리즘 부재로 인한 문제들

### 3-1. 단순 랜덤/최신순 정렬의 한계

#### 현상

초기에는 피드 정렬이 "랜덤"과 "최신순" 두 가지만 존재했다.

- 랜덤: 좋아요 수와 무관하게 완전 무작위 -> 좋아요를 받아도 의미 없음, 투표 동기 부여 실패
- 최신순: 늦게 올린 사진만 상단 노출 -> 먼저 올린 참가자에게 불리

투표 활성화라는 플랫폼 핵심 목표를 달성하지 못하는 구조였다.

#### 해결

Wilson Score + Time Decay + Exploration Bonus + Random Jitter를 조합한 추천 알고리즘을 도입했다. (상세: [feed-ranking-algorithm.md](feed-ranking-algorithm.md))

- 좋아요 많은 사진은 상위에, 적은 사진도 발견 기회 제공
- 새 사진 초기 노출 부스트
- 세션별 랜덤 변동으로 매번 다른 피드

### 3-2. 단시간 투표 급등 (어뷰징)

#### 현상

특정 참가자가 단톡방 등에 링크를 공유하여 단시간에 투표를 집중적으로 받는 현상이 발생했다. 실제로 한 참가자의 사진이 업로드 1시간 만에 시간당 31표를 기록했다. 전체 평균이 시간당 1.7표인 상황에서 약 18배에 달하는 비정상적 속도였다.

이대로 두면 조직적 투표 동원으로 순위가 왜곡되고, 일반 참가자들의 투표 의욕이 떨어질 수 있었다.

#### 해결

Velocity Penalty (2단계 임계값)를 도입했다.

- 시간당 10표 이하: 정상, 패널티 없음
- 시간당 10~20표: 선형 감쇠 (최대 50% 감소)
- 시간당 20표 초과: 로그 스케일 감쇠 (최대 83% 감소)

시간이 지나면 velocity가 자연히 낮아져 패널티가 해제되므로, 자연스럽게 쌓인 투표는 정당하게 반영된다.

### 3-3. 비인기 사진의 상단 고착

#### 현상

Exploration Bonus만 적용했을 때, 투표를 거의 못 받은 사진이 "투표가 적으니까 보너스"를 계속 받아서 상단에 고착되는 문제가 발생했다. 유저 입장에서는 관심 없는 사진이 계속 위에 보이는 좋지 않은 경험이었다.

#### 해결

Quality Gate를 도입하여, 충분한 시간이 지났음에도(6시간 초과) 평균 투표의 30% 미만인 사진은 Exploration 보너스를 최대 70%까지 감쇠시켰다. 새 사진은 6시간 동안 면제하여 초기 기회를 보장하되, 이후에도 투표를 받지 못하면 서서히 하단으로 밀려나도록 했다.

### 3-4. 동일한 피드 순서 (개인화 부재)

#### 현상

모든 유저가 같은 시간에 접속하면 거의 같은 순서의 피드를 보게 되었다. 이미 투표한 사진이 여전히 상단에 위치하여 아직 투표하지 않은 사진을 발견하기 어려웠다.

#### 해결

Voted Demote를 도입하여, 이미 투표한 사진의 점수를 60% 감소시켰다. 유저마다 투표 이력이 다르므로 같은 시간에 접속해도 서로 다른 피드를 보게 되며, 아직 투표하지 않은 사진이 자연스럽게 상단으로 올라온다.
